# План масштабирования: от одной группы к городскому уровню

## Что есть сейчас (исходное состояние)
Бот-дневник ДЗ на **aiogram 2.x** + **Supabase (Postgres через REST)**.
Две таблицы: `homeworks`, `users`. Логика «плоская»:
- ДЗ в одной куче, без привязки к группе — видят все.
- `users.subgroup` (1/2) хранится, но в выборке ДЗ не используется.
- Прав в add/delete нет вообще — любой может добавлять и удалять.
- В `add_homework.py` зашит `GROUP_ID = 427` — группа в API расписания ОмГТУ.
- FSM в `MemoryStorage`, плюс `temp_users = {}` в памяти процесса.

## Целевое состояние
Один бот обслуживает много заведений и групп города. Каждый видит/редактирует
ДЗ только своей группы. Присоединение — по инвайт-ссылкам. Права — по иерархии.

## ИЕРАРХИЯ РОЛЕЙ (по задумке владельца)
Глобальные роли (хранятся в `users.global_role`):
- **admin** — владелец бота (ты). Всё, везде.
- **moderator** — помогает админу глобально.
Роли внутри группы (хранятся в `group_members.group_role`):
- **owner** — староста, админ своей группы.
- **helper** — помощник старосты: добавляет/удаляет ДЗ.
- **member** — обычный пользователь: только смотрит.
Один человек может иметь разные роли в разных группах -> роли в группе вынесены
в отдельную таблицу-связку `group_members`, а не в `users`.

## ДВА ТИПА ПРИГЛАСИТЕЛЬНЫХ ССЫЛОК (таблица `invites`)
1. **create_group** — одноразовая. Ты выдаёшь старосте лично.
   По ней человек создаёт группу (вводит название, привязка к вузу) и
   становится её owner. Право «создать группу» НЕ выдаётся всем подряд.
2. **join_group** — многоразовая (или с лимитом/сроком). Староста генерирует
   в своей админке и кидает в чат группы — одногруппники входят как member.
Так владельцу не нужно касаться каждой группы руками: он раздаёт только
ссылки типа 1 (по одной на группу), дальше староста всё делает сам.

## Модель данных (целевая) — РЕАЛИЗОВАНО в models.py
- institutions(id, name, city, schedule_provider, schedule_provider_id)
- groups(id, institution_id, name, external_schedule_id, owner_user_id)
- users(id, user_id, username, first/last_name, active_group_id, global_role, subgroup)
- group_members(id, group_id, user_id, group_role, subgroup, joined_at)
- invites(id, code, invite_type, group_id, institution_id, created_by,
          is_single_use, max_uses, uses_count, is_active, expires_at)
- homeworks(id, group_id, subject, task, date_for, subgroup, created_by, attachments)

Ключевая идея: выборка ДЗ всегда `WHERE group_id = активная группа пользователя`.
`external_schedule_id` у группы заменяет зашитый GROUP_ID=427.

## Принципы
1. Каждый этап оставляет бота рабочим.
2. Сначала данные, потом логика.
3. Данные текущей группы не теряем -> «группа №1».
4. Внешний API расписания — деталь заведения, не всего бота.
5. Глобальные роли в users; роли в группе — в group_members.
